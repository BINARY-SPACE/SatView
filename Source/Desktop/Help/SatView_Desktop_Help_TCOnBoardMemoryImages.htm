TC On-Board Memory Images Display

SUMMARY

Saturday, October 10, 2015
Version 1.0

The TC On-Board Memory Images Display visualizes the state and contents of the on-board processor memory areas. Dumps are detected automatically (on demand) and are updated accordingly in the database. Customizable versioning algorithms allow an effective management of those dumps and an integrated analyzing tool facilitates the comparison of (dumped) memory images. Help is also provided with the generation of the telecommands necessary to uplink memory images or to request a dump from on-board memory.

How do I...

Open the TC On-Board Memory Images Display

  1. Push the Memory Images toggle-button inside the On-Board Services panel of the Telecommand ribbon bar category or, alternatively, open the Windows button's menu inside the Window panel of the System ribbon bar category and select the TC On-Board Memory Images item.
  2. Optionally configure the TC On-Board Memory Images display.

  • Understand the Layout

  1. Open the TC On-Board Memory Images Settings Dialog Box if it is not open yet:

    Managing Dumps

    In order to synchronize the representation of the on-board memory images, occurring dumps have to be archived on-ground. This can be achieved by specifying how that data is processed and maintained.

    1. Select the Dumps page.

    2. Updating Database Components

      SatView™ maintains two levels of databases: A Database Management System (DBMS) hosting the database and a binary file for fast access which is a compiled excerpt of the DBMS contents.

      1. Choose the appropriate options:

        Option

        Description

        Update Database Management System (DBMS)

        Every dump is stored in the DBMS according to strategies defined by the specified versioning.
        Note:
        It is recommended to enable this option because it serves as a backup for the binary file.

        Update Binary File

        Downloaded on-board memory data is inserted into the binary database file.
        Note:
        If this option is not enabled and only the DBMS is updated, the DBMS has to be re-compiled to make the new dumps accessible.

        Note:
        If neither of the above options is selected, the most recent dumps will be available only.

      2. Specify the delay n between the time when a dump is received and the update of the selected database components by adjusting the Use an update delay of n seconds spin box.
        Note:
        The delay of n seconds should not be too short in order to enable bulk transfers to the databases which is more efficient than updating each dump individually.

      Customizing the Dump Versioning

      SatView™ implements a sophisticated versioning mechanism in order to simplify the management of on-board memory image dumps.

      1. Inside the Versioning group box select the on-board processor for which the dump management should be modified in the Processor combo box.
      2. Choose the appropriate versioning strategy:

        Strategy

        Description

        None

        Only the most recent dump is stored. Its name is identical to the one specified for the memory image template required to manage the corresponding dump.

        Use a simple increasing number

        Enumerates the dumps in a way that can be further specified:

        1. Enter a prefix x into the With Prefix: x edit field. The prefix x will be precluded to the sample number.
        2. Limit the number of samples n to be maintained in a cyclic way by specifying it in the Limit to: n cyclic dumps spin box.

        Time interval is bigger than n minutes

        Uses a major.minor numbering where the major number is increased when the time interval between the dumps is bigger than n minutes. The version number can be precluded with a prefix x and limited to n samples:

        1. Enter a prefix x into the With Prefix: x edit field. The prefix x will be precluded to the version number.
        2. Limit the number of samples n to be maintained in a cyclic way by specifying it in the Limit to: n cyclic dumps spin box.

        Data changes more than n %

        Uses a major.minor numbering where the major number is increased whenever the data changes (on a byte basis) between the dumps are bigger than n percent. The version number can be precluded with a prefix x and limited to n samples:

        1. Enter a prefix x into the With Prefix: x edit field. The prefix x will be precluded to the version number.
        2. Limit the number of samples n to be maintained in a cyclic way by specifying it in the Limit to: n cyclic dumps spin box.

    Changing the Logging Options

    Every change with respect to the operational configuration of the on-board memory images management can be logged and reviewed at a later time. This allows the reconstruction of any configuration changes.

    1. Select the Logging page.

    2. Configuring the Options

      1. Push the Options... button. The TC On-Board Memory Images Logging Options Dialog Box will appear.
      2. Type the name for the log file into the Filename edit field. Use the Browse... button in order to locate the file if necessary.
        Note:
        The file name can be identical to those of the other log sources like, for example, command sources. This may be interesting if only one single log file should be generated.
      3. Check one of the available file policy options:

        Option

        Description

        Overwrite logs as needed

        Enables the cyclic overwrite mode.
        Note:
        As soon as the log file reaches the specified size or no more space is available on the storage media new logs overwrite the oldest ones.

        Overwrite logs older than n days

        Enables the cyclic overwrite mode.
        Note:
        Logs are overwritten by new ones as soon as they are older than n days. This can happen even before the specified file size is reached.

        Do not overwrite any logs

        Disables the cyclic overwrite mode.
        Note:
        Stops logging when either the specified size is reached or no more space is available on the storage media.
        Optional:
        • Mark the Disabled check box in order to prevent the log file from being updated.
      4. Check one of the available file size options:

        Option

        Description

        Unlimited

        Archives logs as long as space is available on the storage media.

        Limited to n KBytes

        Limits the file size to n KB.
        Optional:
        • Mark the Reset check box in order to reset the log file.
      5. Push the Retrievals... button which opens the TC On-Board Memory Images Log Retrieval Options Dialog Box and type the name of the log file to be used for retrievals into the Filename edit field. Use the Browse... button in order to locate the file if necessary.
        Push the OK button.
        Note:
        The file name is usually identical to the one specified for logging but can also be identical to those of the other log sources like, for example, command sources. This may be interesting if only one single file is used for the archiving of all those log sources.
      6. Push the OK button.

Explore Features

  • Each of the on-board processors has some memory allocated to it which can be dumped or patched on request. It is vital that a correct and up-to-date representation of the on-board memory configuration can be maintained automatically and that there is support available to manipulate those buffers through telecommands.
    In the following context, Memory Areas are nothing else than a description about the configuration of a certain memory area allocated to an on-board processor. They can be overlayed with Memory Dumps or Memory Images respectively which represent the data dumped from or the data to be patched to that location on-board.
    Read more about:

    Memory Areas

    Memory Areas are a descriptive representation of a memory area on-board. They can be used to create patches or to interpret dumps more easily.

    1. Activate the on-board processor containing the memory area of interest in the processor directory list.
    2. Select the Memory page.

    3. Browsing the Memory Area Configuration

      • Choose the desired memory area in the Name combo box.
        The following information will then be displayed:
        • A short explanation about the function of the memory area.
        • Some attributes of the memory area:

          Attribute

          Description

          Patchable

          Indicates that the memory area can be patched.

          Dumpable

          Indicates that the memory area can be downloaded.
          Additional update flags:

          Update Flag

          Description

          Automatic

          Allows an automatic update of the memory area data in the database as specified in 'Managing Dumps'.

          Allow inhibition of automatic updates

          Allows a temporary suspension of memory area data updates in the database.
        • A table describing the structure of the memory area.

      Mapping Memory Images or Memory Dumps

      Memory images or memory dumps can be mapped over memory areas. This simplifies the preparation of patches or the interpretation of dumps because the memory area structure is directly associated with the corresponding data.

      1. Choose the desired memory area in the Name combo box.
      2. Select the memory image or memory dump to be mapped in the Image combo box.
      3. Specify the version of the memory image in the Version combo box.
        The data of the chosen memory image is then mapped onto the appropriate memory area locations inside the Value column.

      4. Preparing & Performing Patches

        1. Enter a unique version name into the Version edit field.
          Note:
          It is strongly recommended to use different version names (or prefixes) for patches and dumps. This prevents any unwanted overwrites.
        2. Fill the appropriate data in the Value column of each table row. This can be achieved by double-clicking onto the Value field and then entering the data.
          Notes:
          • Illegal data is either blocked with an audible alert or indicated as such through background coloring and a message above the Properties group box.
          • The Undo and Redo buttons at the lower left corner can be used to undo/redo any input.
        3. Push the Transact button in order to store the memory image in the database. If required, the memory area can now be patched with the created memory image.

        4. Patching the Memory Area

          1. Push the Patch... button. The TC On-Board Memory Images Patch Properties Dialog Box will appear which contains all telecommands already initialized with the data and necessary to patch the memory area.
          2. Specify the on-board execution mode by checking either the Immediately or At Time radio button. In the latter case the future on-board execution time has to be entered into the associated time spin-box.
          3. Choose the command source from which the prepared telecommand should be dispatched from by checking either the Manual Stack or Automatic Stack radio button.
          4. Decide if the telecommand has to be released immediately or at a specific time t by selecting the Immediately or At Time radio button and eventually entering the time t.
          5. Mark the Pre-Authorized check box in order to force an automatic dispatch from the chosen command source.
          6. Push the OK button.

          Note:
          This option is not available if the memory area is not patchable i.e. the option Patchable is not enabled.

        Interpreting Dumps

        • Each row of the table represents an on-board memory entity and its value can be found in the Value column.
        • Additional details about those values can be obtained by using the tool-tip feature.

    Memory Images

    Memory Images contain the data to be patched to the associated memory area on-board. They can be created from scratch or derived from a previous dump of that memory area.

    1. Activate the on-board processor containing the memory area of interest in the processor directory list.
    2. Select the Images page.

    3. Creating a Memory Image Template

      A memory image template is a prerequisite for managing on-board memory images and memory dumps. It defines the source and the address space to be managed.

      1. Enter a unique name for the memory image into the Name edit field.
      2. Select the TEMPLATE item inside the Version combo box.
      3. Type the start address for the memory image into the Address column of the table.
        Use the format: XXXXXXXX:XXXXXXXX where X is a hexadecimal digit.
      4. Specify the size of the memory image into the Size column.
        Note:
        The size must be specified in data entities (not bytes) equivalent to the data width.
      5. Choose the appropriate attributes:

        Attribute

        Description

        Patchable

        Indicates that the memory image with the specified name can be patched to the corresponding memory area on-board.

        Dumpable

        Indicates that the corresponding memory area can be dumped and managed under the name of the defined memory image.
        Select additional update flags:

        Update Flag

        Description

        Automatic

        Allows an automatic update of the memory image data in the database as specified in 'Managing Dumps'.

        Allow inhibition of automatic updates

        Allows a temporary suspension of memory image data updates in the database.
      6. Push the Transact button.
        This updates the database components accordingly.

      Deleting a Memory Image Template

      1. Select the name of the memory image to be deleted in the Name combo box.
      2. Choose the TEMPLATE item in the Version combo box.
        Note:
        The template does not exist any more if memory images have been defined basing on that template. Deleting all those memory image versions automatically removes the associated memory image template.
      3. Push the Delete button.
        This updates the database components accordingly.

      Creating a new Memory Image

      1. Choose the memory image for which a new version should be created in the Name combo box.
      2. Select either the TEMPLATE item or any other old version of the memory image in the Version combo box.

      4. Type the data in a hexadecimal format into the Data column.

        • Illegal data is either blocked with an audible alert or indicated as such through background coloring and a message above the Properties group box.

        This updates the database components accordingly.

    Memory Dumps

    Memory Dumps contain the data that was downloaded from the associated memory area on-board. They can be used to derive a modified memory image which is then patched back to that memory area. Some other features to analyze dumps are also available.

    1. Activate the on-board processor containing the memory area of interest in the processor directory list.

    2. Browsing a Memory Dump

      As soon as a memory image template is created the corresponding memory dumps are handled automatically. The number of dumps maintained, versioning and many other attributes can be configured.

      1. Select the Dumps page.
      2. Choose the memory area of which a dump needs to be browsed in the Name combo box.
      3. Select the dump of interest in the Version combo box.
        The memory dump data is now displayed in the table below.
        Note:
        If more than one memory dump originating from the same memory area is available, the < (step backwards in time) or > (step forward) buttons can be used to navigate through those dumps.

      When awaiting a memory dump proceed as follows:

      1. Select the Dumps page.
      2. Choose the memory area from which a dump is expected to arrive in the Name combo box. It does not matter if a previous dump is currently visible because it will be updated automatically.
      3. Mark the Show downloaded data check box if the downloaded data should be displayed immediately after reception without waiting for the dump to be complete.

      Read more about:

      Memory dumps can be exported in a readable format for use with other applications.

      1. Decide if the selected memory dump should be saved in an ASCII or Unicode encoding by making the appropriate use of the Unicode check box.
      2. Push the Save as... button and enter the file name for the memory dump to be saved into the dialog box popping-up.

      Analyzing a Memory Dump

      Comparing memory images with memory dumps or memory dumps versus other dumps can be an important capability, for example to check if a selected on-board memory area contains the expected data.

      1. Select the Analysis page.
      2. Choose the memory image to be analyzed in the Name combo box on the left side.
      3. Select the requested issue of that image by selecting it in the Version combo box below.
      4. Perform the same actions on the right side i.e. choose the memory image or dump to which the one on the left side should be compared to in the Name and Version combo boxes.
      5. Push the VERSUS button. This will generate a list of all differences between the compared memory images or dumps respectively.
        Note:
        Enable the Show all results option to show the comparison results for all memory area entities, regardless if they are equal or not.

      6. Patching an analyzed (partial) Memory Image

        The results of analyzed memory images or dumps can be patched i.e. only the different memory area entities are uploaded. This may be an efficient approach to synchronize on-board memory areas.

        1. Push the Patch... button. The TC On-Board Memory Images Patch Properties Dialog Box will appear which contains all telecommands already initialized with the data and necessary to patch the memory area.
        2. Specify the on-board execution mode by checking either the Immediately or At Time radio button. In the latter case the future on-board execution time has to be entered into the associated time spin-box.
        3. Choose the command source from which the prepared telecommand should be dispatched from by checking either the Manual Stack or Automatic Stack radio button.
        4. Decide if the telecommand has to be released immediately or at a specific time t by selecting the Immediately or At Time radio button and eventually entering the time t.
        5. Mark the Pre-Authorized check box in order to force an automatic dispatch from the chosen command source.
        6. Push the OK button.

        Note:
        This option is only available if partial patches of on-board memory are supported by the destination sub-system and when the corresponding memory area is flagged as 'patchable'.

        Saving an analyzed Memory Image or Memory Dump

        The results of analyzed memory images or dumps can be exported in a readable format for use with other applications.

        1. Decide if the analysis results should be saved in an ASCII or Unicode encoding by making the appropriate use of the Unicode check box.
        2. Push the Save as... button and enter the file name for the results to be saved into the dialog box popping-up.
        3. Push the OK button.

        Printing an analyzed Memory Image or Memory Dump

        The results of analyzed memory images or dumps can also be printed.

        • Push the Print button.
          This prints the analyzing results currently available.

        Note:
        The Default print profile is applied when printing this way.

      Printing a Memory Dump

      • Push the Print button.
        This prints the memory dump currently selected.

      Note:
      The Default print profile is applied when printing this way.

      Deleting a Memory Dump

      1. Select the Dumps page.
      2. Choose the memory dump in the Name combo box.
      3. Select the version of the memory dump to be deleted in the Version combo box.
      4. Push the Delete button.
        This updates the database components accordingly.

      Stopping an on-going Memory Dump

      Because some memory dumps may be very large in size, the capability to abort a requested or on-going dump should be supported as well.

      1. Select the Dumps page.
      2. Choose the memory image for which the requested or on-going dump should be stopped in the Name combo box.
      3. Push the Stop Dump... button. The TC On-Board Memory Image Stop Dump Properties Dialog Box will appear.
      4. Specify the on-board execution mode by checking either the Immediately or At Time radio button. In the latter case the future on-board execution time has to be entered into the associated time spin-box.
      5. Choose the command source from which the prepared telecommand should be dispatched from by checking either the Manual Stack or Automatic Stack radio button.
      6. Decide if the telecommand has to be released immediately or at a specific time t by selecting the Immediately or At Time radio button and eventually entering the time t.
      7. Mark the Pre-Authorized check box in order to force an automatic dispatch from the chosen command source.
      8. Push the OK button.

      Note:
      This option is not available if the destination sub-system does not support it.

    Whenever the TC On-Board Memory Images display needs to be printed in a different way than defined by the Default print job profile this method is the one to be used. It allows the modification of the print properties without changing the profile.

    Event-driven Printing

    Sometimes it may be advantageous to print the TC On-Board Memory Images display automatically at a certain time, when an on-board memory dump occurs or when a predefined telemetry event occurs. This can be achieved by scheduling a print job.
